# 🧪 אסטרטגיית הבדיקות

> העיקרון המנחה: **ה-RLS הוא המוצר.** דליפת מידע אחת של ילד שווה יותר מכל הבאגים האחרים יחד — לכן פירמידת הבדיקות כאן הפוכה מהרגיל: הבסיס הוא בדיקות DB, לא בדיקות UI.

## 1. שכבות הבדיקה

| שכבה | כלי | מתי רץ | בעלים |
|-------|-----|---------|--------|
| **RLS ופרטיות** (הקריטית) | pgTAP ‏(`supabase test db`) | כל שינוי DB + CI על כל PR | Antigravity |
| לוגיקת DB (מנוע, state machine) | pgTAP | כל שינוי בפונקציות | Antigravity |
| Types + lint | `tsc --noEmit`, ‏`expo lint` | כל commit + CI | שניהם |
| Unit ‏(UI utils: phone, geo, i18n) | Jest | CI | Cursor |
| E2E ידני מובנה | צ'קליסט (סעיף 4) | לפני כל אבן דרך + לפני build | שניהם |
| Smoke ל-Edge Functions | סקריפט curl מול סביבה מקומית | שינוי בפונקציה | Antigravity |

הערה: `packages/matching` + בדיקות ה-Jest שלו מבוטלים עם איחוד מקור האמת (ARCHITECTURE_REVIEW H2) — הלוגיקה נבדקת ב-pgTAP מול הפונקציה האמיתית.

## 2. חוקי הזהב לבדיקות RLS

1. **כל policy נבדקת לשני הכיוונים** — מי שמורשה רואה, ומי שלא — חסום. בדיקת חסימה חשובה יותר.
2. **מטריצת התוקפים** — לכל משאב רגיש נבדקים לפחות: הורה אחר · משלבת ללא קשר · משלבת עם בקשה pending (לא approved!) · משלבת עם match שהסתיים · אנונימי.
3. **תרחישי ההסלמה מהביקורת** חובה בסוויטה: משלבת מנסה self-approve ‏(C1) · משלבת קוראת עמודות מוסתרות מ-children ‏(C2) · כתיבת audit בכל קריאת TIER 2+ ‏(C3) · לא-אדמין קורא RPC אדמיני.
4. בדיקה נכשלת = אין merge. בלי יוצאים מן הכלל.

## 3. בדיקות state machine (pgTAP)

לכל מעבר ב-match_request: המעבר החוקי מצליח לתפקיד הנכון, וכל מעבר לא חוקי (סטטוס שגוי / תפקיד שגוי / כפילות) נכשל עם שגיאה. לפחות 12 בדיקות: 6 מעברים × 2 כיוונים.

## 4. צ'קליסט E2E ידני (לפני כל אבן דרך)

סביבה: מקומית או ענן-staging, שני מכשירים/סימולטורים (הורה + משלבת), עברית RTL.

- [ ] הרשמת הורה → פרופיל ילד מלא → פרסום (D6 מוצג נכון)
- [ ] הרשמת משלבת → מסמכים → אימות (דרך Admin-1 או Studio) → push "מאומתת"
- [ ] הורה רואה התאמות עם הסברים; אין ציון מספרי (D3)
- [ ] שליחת בקשה → push למשלבת → מכתב → push להורה → LetterCard
- [ ] אישור עם sheet הגילוי → פרטי היכרות → "התחלנו לעבוד" → match פעיל
- [ ] check-in בתוך/מחוץ ל-geofence — שתי התוצאות נכונות
- [ ] שאלון יומי ≤60 שניות → סיכום AI מופיע להורה → push
- [ ] revoke גישה → המשלבת מאבדת גישה מיידית (לבדוק בפועל!)
- [ ] סיום match → דירוג עיוור (D14)
- [ ] כל המסע באנגלית LTR — סריקה ויזואלית
- [ ] מצב offline: פתיחת מסכים מרכזיים + ניסיון כתיבה

## 5. CI (GitHub Actions — להקים באבן דרך 0)

Workflow על כל push/PR: ‏(1) ‏supabase start + מיגרציות מאפס + ‏`supabase test db` ‏(2) ‏tsc + lint ‏(3) Jest. אדום = לא ממשיכים.

## 6. נתוני בדיקה

`supabase/seed.sql` מחזיק: 2 הורים (אחד עם 2 ילדים), 4 משלבות (מאומתת, בהמתנה, נדחתה, לא-הגישה), בקשות בכל סטטוס, match פעיל עם שבוע יומנים, ואדמין. כל תרחיש E2E ניתן להרצה מיידית אחרי `supabase db reset`.
