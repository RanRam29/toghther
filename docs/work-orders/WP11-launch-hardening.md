## 1. הגבלת קצב (Antigravity)

### 1.1 שכבת ההתחברות (תצורה, לא קוד)
ב-`supabase/config.toml` תחת `[auth.rate_limit]` (ולוודא שהערכים תואמים גם בלוח הבקרה בענן — התצורה המקומית לא מסונכרנת אוטומטית):
- `sms_sent = 10` לשעה ‏· `token_verifications = 30` ל-5 דקות ‏· `sign_in_sign_ups = 30` ל-5 דקות ‏· `email_sent = 4` לשעה.
- צד Twilio: לוודא Geo-Permissions לישראל בלבד + התראת חריגת תקציב (הגנת SMS-pumping).

### 1.2 שכבת ה-RPC — מנגנון אחיד
טבלה `rpc_rate_limits(user_id, action, window_start, count)` + פונקציית עזר:
```sql
public.enforce_rate_limit(p_action text, p_max int, p_window interval) RETURNS void
-- SECURITY DEFINER + search_path; חריגה => RAISE 'Rate limit exceeded for %'
```
לשלב בתחילת: `hide_match_profile` (20/יום) · `mark_day_off` (10/יום) · `invite_secondary_parent` (5/יום) · `set_child_field_visibility` (60/יום) · `log_screenshot_event` (30/יום). ניקוי חלונות ישנים ב-cron היומי הקיים. pgTAP: הקריאה ה-N+1 נכשלת, משתמש אחר לא מושפע.

## 2. צמצום CORS בפונקציות הענן (Antigravity)
כיום `CORS_ORIGIN` עם ברירת מחדל `*`. לשנות בכל ארבע הפונקציות: קריאת רשימת מקורות מופרדת בפסיקים מ-`CORS_ORIGINS`, החזרת ה-Origin הפונה רק אם הוא ברשימה (אחרת ללא כותרת). לקבוע secret בענן: הדומיין של Expo-web + סכמת האפליקציה. לתעד ב-`docs/DEPLOYMENT.md`.

## 3. גיבויים ושחזור (Antigravity + בעל המוצר)
1. לוודא בלוח הבקרה: גיבוי יומי פעיל; אם התוכנית מאפשרת — להפעיל PITR.
2. **תרגול שחזור** (חובה, לא תיאורטי): שחזור הגיבוי האחרון לפרויקט Supabase זמני → הרצת `supabase test db` מולו → תיעוד משך ותוצאה ב-`docs/DEPLOYMENT.md` סעיף חדש "נוהל שחזור". תרגול חוזר רבעוני.
3. לוודא שה-bucket ‏`documents` כלול במנגנון הגיבוי (Storage לא תמיד כלול בגיבוי ה-DB!) — אם לא, סקריפט סנכרון שבועי לאחסון מוצפן נפרד.
