# 🧡 מסע ההורה — צעד-צעד עם חוזי נתונים

> מרחיב את PRODUCT_UX_SPEC חלק 2. כל צעד: מה המשתמש עושה → מה המערכת קוראת/כותבת → לאן ממשיכים. מזהי מסכים מפנים ל-[05-SCREENS.md](05-SCREENS.md).

## שלב א' — כניסה והקמה

### P-01: התקנה ראשונה → login
1. splash → `S-AUTH-01` (login). קלט: טלפון → OTP (‏`S-AUTH-02`).
2. **כתיבה:** auth.users נוצר; טריגר `handle_new_user` יוצר profile (role מ-metadata).
3. משתמש חדש → `S-AUTH-03` (בחירת תפקיד) → הורה → onboarding. משתמש חוזר → ניתוב לפי role (route guard קיים).

### P-02: פרופיל ילד (onboarding)
צעדים (מסך לכל צעד, שמירת טיוטה ב-zustand + persist אחרי כל צעד):
1. שם פרטי + גיל → 2. מסגרת + אזור → 3. שעות ליווי נדרשות → 4. קטגוריית צורך + רמת תפקוד + תקשורת ורבלית (עם משפט ההרגעה — ראו קופי) → 5. "מה עובד / מה מקשה / מה הניצחון" (free-text מונחה) → 6. מסך פרסום (D6).
- **כתיבה בסיום:** `children` (שדות TIER 0–1, `published` לפי בחירה) + `child_details` (שדות רגישים). שתי כתיבות בטרנזקציה לוגית אחת.
- **מקרה קצה:** נטישה באמצע → בכניסה הבאה ממשיכים מהצעד האחרון ("נמשיך מאיפה שעצרנו?").
- ילד נוסף: אותו flow מ"הוסף ילד" בבורר הילדים.

## שלב ב' — מציאה ובקשה

### P-03: מסך התאמות (`S-PAR-01`)
- **קריאה:** `get_matches_for_child(child_id, 5)` דרך Edge `calculate-matches`. הסבר AI מוצג מה-cache (ARCHITECTURE_REVIEW M1); עד אז — `match_reason` מה-DB.
- ריק → מצב waitlist (ראו S-PAR-01 states).
- לחיצה על כרטיס → `S-PAR-02` (פרופיל משלבת).

### P-04: פרופיל משלבת (`S-PAR-02`) → שליחת בקשה (`S-PAR-03`)
- **קריאה:** `professionals` (policy ‏public_read) + `reviews` (parent_browse) — רק אם ≥3 (D4).
- CTA "שלח/י בקשת היכרות" → sheet "מה ייחשף עכשיו" (TIER 1) → הודעה אישית אופציונלית → שליחה.
- **כתיבה:** `match_requests` insert ‏(status=pending, initiated_by='parent'). אכיפת מכסה D7 (5 פתוחות/ילד) — בצד DB (constraint/RPC) + חיווי ב-UI.
- **Push למשלבת** (מטריצה, אירוע 1).

### P-05: מעקב בקשות (`S-PAR-04`)
- **קריאה:** `match_requests` של ילדי ההורה + סטטוס. Realtime subscription לעדכון חי.
- interested → מציג LetterCard + כפתור "אשר/י והמשיכו להיכרות".
- rejected → "לא זמינה כרגע" + הצעת חלופות. expired → שקט, עם חלופות.

## שלב ג' — אישור והפעלה

### P-06: אישור (מעבר TIER 2) — `S-PAR-05`
1. sheet גילוי מלא: רשימת מה ייחשף (שם מלא, אבחנה, מה עובד/מקשה, פרטי קשר) + אישור מפורש.
2. **כתיבה:** RPC ‏`approve_request(request_id)` — מעביר ל-`approved`/**TIER 2 בלבד**. ⚠️ לפי D10 ותיקון H4: אישור **אינו** יוצר match פעיל; ה-match נוצר רק ב-P-07. (המימוש הנוכחי יוצר match מיד — מתוקן ב-WP1.)
3. → `S-PAR-06` "פרטי היכרות": כרטיס המשלבת + חיוג/וואטסאפ + הנחיית "קבעו שיחה ואז מפגש עם הילד".

### P-07: הפעלת match — מ-`S-PAR-06`
- "התחלנו לעבוד יחד! 🎉" → RPC ‏`create_match_from_request` → מעבר ל-`S-PAR-07` (מסך match).
- "לא התאים" → RPC ‏`withdraw_request` עם reason → חזרה להתאמות, המכסה משוחררת.

## שלב ד' — ליווי שוטף

### P-08: מסך match פעיל (`S-PAR-07`)
- **קריאה:** `matches` + ‏`checkins` של היום + ‏`daily_logs` (עמוד אחורה בגלילה) + סיכומי AI.
- ראש: סטטוס נוכחות היום. גוף: פיד יומן + גרף מגמה שבועי. תחתית: "מי צפה בתיק" (audit — תלוי C3), "מה דנה רואה" (הרשאות).
- Push נכנסים: check-in (ניתן לכיבוי), סיכום יומי מוכן.

### P-09: ניהול הרשאות ו-revoke (`S-PAR-08`)
- מתגי חשיפה פר-שדה של child_details + "השהה גישה" (מעביר match ל-paused) + "סיים עבודה משותפת" (ended + reason).
- **כתיבה:** RPC ייעודיים; revoke נכנס לתוקף מיידית (RLS נשען על status).

### P-10: סיום ודירוג (`S-SHARED-01`)
- 24 שעות אחרי ended → push. דירוג: 3 קריטריונים + טקסט. עיוור לפי D14.
- **כתיבה:** `reviews` insert; טריגר מעדכן rating_avg.

## מקרי קצה רוחביים
| תרחיש | התנהגות |
|--------|----------|
| הורה מוחק חשבון | cascade מוחק ילדים ובקשות; אצל משלבות עם בקשה פתוחה הכרטיס נעלם ("הכרטיס הוסר") |
| שני ילדים, אותה משלבת | מותר — שני matches נפרדים; היומנים נפרדים |
| ההורה מנסה בקשה 6 לילד | חסימה רכה: "הגעת ל-5 בקשות פתוחות — המתן לתשובה או משוך בקשה" |
| ילד עובר מסגרת/כתובת | עריכת פרופיל ילד → ההתאמות מחושבות מחדש; matches פעילים לא נפגעים |
