# 💜 מסע המשלבת — צעד-צעד עם חוזי נתונים + runbook אדמין

> מרחיב את PRODUCT_UX_SPEC חלק 3. מזהי מסכים — [05-SCREENS.md](05-SCREENS.md).

## שלב א' — הקמה ואימות (שער ה-supply — הכי חשוב עסקית)

### F-01: הרשמה ופרופיל מקצועי
1. login/OTP (משותף) → בחירת תפקיד "משלבת" → onboarding מקצועי.
2. צעדים: שם תצוגה + תמונה (חדש — חובה מוצרית, R4) → התמחויות (ChipSelect מ-need_category) → מסגרות + ניסיון → זמינות שבועית (בורר ימים/טווחי שעות) → אזור + רדיוס (ברירת מחדל 15 ק"מ) → ביו מונחה ("ספרי על הגישה שלך…").
3. **כתיבה:** `professionals` insert (verified='pending').

### F-02: העלאת מסמכים (`S-PRO-02`)
- checklist: ת"ז, תעודת יושר, תעודת הכשרה (+ "אחר" אופציונלי). צילום/קובץ.
- **כתיבה:** upload ל-bucket ‏`documents` (private!) → `document_uploads` insert. עם שלושת החובה → verified='submitted'.
- כל פריט עם סטטוס עצמאי: הועלה / באימות / אושר ✓ / נדחה + סיבה.

### F-03: מצב "ממתינה לאימות" (`S-PRO-03`)
- אפליקציה נעולה-חלקית: עריכת פרופיל ✓, צפייה בילדים/בקשות ✗.
- מסך בית זמני: progress המסמכים + "בדרך כלל עד 2 ימי עסקים" + מה אפשר בינתיים (להשלים ביו/תמונה).
- push בהכרעה: אושרה → "את מאומתת ✓"; נדחה מסמך → מה בדיוק לתקן. **זהו רגע הנטישה המסוכן — כל שיפור כאן שווה יותר מכל feature.**

## שלב ב' — קבלת עבודה

### F-04: בית — בקשות נכנסות (`S-PRO-01`)
- **קריאה:** `match_requests` שלה (policy קיימת) + נתוני TIER 1 של הילד (אחרי C2: דרך RPC מדורג). Realtime לבקשות חדשות.
- כרטיס בקשה: אווטאר (D19), שם פרטי + גיל, קטגוריה, אזור כללי (D2), שעות, הודעת ההורה אם יש.
- פעולות: "מעוניינת — כתבי מכתב" / "לא זמינה" (+ סיבה אופציונלית).

### F-05: כתיבת המכתב (`S-PRO-04`) — ה-friction המכוון (R3)
- פרומפט: "ספרי בקצרה: מי את ולמה את מתאימה דווקא לילד הזה? 3–5 משפטים". דוגמה מוצלחת מתקפלת. 100–500 תווים.
- **כתיבה:** RPC ‏`respond_to_request(id, 'interested', cover_letter)` (אחרי C1).
- אחרי שליחה: "המכתב נשלח — ההורה יקבל התראה" + חזרה לבית.

### F-06: browse משני (`S-PRO-05`)
- **קריאה:** view ‏`children_tier0` בלבד (אחרי C2). פילטרים: קטגוריה, מרחק (לפי אזור), שעות.
- "אשמח להכיר 💜" → מכתב (אותו flow) → `match_requests` insert ‏(initiated_by='professional', status='pending').
- שפת ציפיות: "ההורה יקבל את פנייתך ויחליט". בלי מונה צפיות, בלי "כמה מתחרות" — לא מייצרים לחץ סביב ילדים.

### F-07: אושרה (TIER 2) → היכרות
- push "ההורה של נ׳ אישר" → מסך פרטי היכרות (מקביל ל-S-PAR-06): פרטי ההורה, חיוג/וואטסאפ, טיפים למפגש היכרות.
- ההפעלה בידי ההורה (D10) — אצלה מוצג "ממתין להפעלה על ידי ההורה" עד שנוצר match.

## שלב ג' — עבודה שוטפת

### F-08: "היום שלי" (`S-PRO-06`)
- **קריאה:** matches פעילים + האם נעשה check-in היום + האם מולא שאלון.
- כרטיס פר-match להיום: "הגעתי — Check-in" (גדול, ראשי) → בקשת הרשאת מיקום ברגע הצורך (D22) → `verify_checkin` RPC → הצלחה ("נרשם ✓ 08:03") או geofence fail עם הסבר ונסיון חוזר.
- אחה"צ הכרטיס מתחלף ל"איך היה היום עם נועם?" → שאלון.

### F-09: השאלון היומי (`S-PRO-07`) — 60 שניות (D11)
- מצב רוח (חובה) → עד 3 סליידרים לפי מדדי ה-match (D12) → הערה אופציונלית → שליחה.
- **כתיבה:** `daily_logs` upsert ‏(match_id, log_date) → טריגר/queue מפעיל Edge ‏`process-daily-log` → ai_summary נכתב → push להורה.
- תזכורת 13:00 אם לא מולא. מילוי רטרואקטיבי — עד 48 שעות אחורה בלבד.

### F-10: מוניטין ופרופיל
- דירוגים (עיוורים, D14) + ביקורות טקסט. מתג "פנויה להחלפה" (backup_available) — תשתית לחמ"ל v1.5.

## חלק ג' — runbook אדמין (MVP, ב-Supabase Studio)

1. **תור:** שאילתה שמורה — `professionals WHERE verified='submitted' ORDER BY updated_at` + המסמכים של כל אחת. SLA: 2 ימי עסקים (D15).
2. **צ'קליסט חובה פר-משלבת:** □ ת"ז תואמת שם □ תעודת יושר בתוקף וללא רישום רלוונטי □ תעודת הכשרה ממוסד מוכר □ תמונת פרופיל סבירה □ ביו ללא פרטי קשר (עוקף פלטפורמה).
3. **הכרעה:** verified='verified' + verified_by + verified_at, או 'rejected' + rejection_note ספציפית לכל מסמך שנפסל. אסור 'rejected' בלי note.
4. **אסור:** למחוק רשומות; לערוך שדות אחרים של המשלבת; לאשר עם מסמך חסר ("הכול או כלום", D15).
5. כל הכרעה שולחת push אוטומטי (טריגר) — האדמין לא שולח הודעות ידנית.
